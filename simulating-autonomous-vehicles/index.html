Simulating Connected & Autonomous Vehicles

September 8, 2020

Why Simulate?

Communication is one of the most important opportunities for self driving cars. If you do not agree, I wrote an article just for you. However, communication is often overlooked due to a variety of implementation challenges.

One such challenge is a lack of a clear value proposition that motivates widespread adoption. Because widespread adoption of V2X technology is expensive, the best way to test value propositions is through traffic simulation.

However, simulating connected and autonomous vehicles is still on the cutting edge of simulation technology and practices. Many connected and automated vehicle simulations exist within custom-built simulation frameworks for academic papers. These simulation frameworks are not easily adaptable by traffic engineers or urban planners.

This is why I created a generalized Python software package that makes it easier to simulate connected and automated vehicle use cases in the leading traffic simulation software, PTV Vissim.

If you want some immediate audio-visual satisfaction, you can check out my technical presentation at the 2019 Transportation Research Board Visualization Symposium in Washington, DC.

Disclaimer: You may be confused to see UAVs in a ground traffic simulator. To learn more about the research objectives behind this presentation, read my article on Cooperative Autonomy. (coming late September)

Simulation Scope

From simulating the chassis dynamics of a single vehicle to simulating the traffic dynamics of an entire city, there is a wide range of simulation purposes and tools. Many autonomous vehicle companies are building custom video game style simulators complete with soft sensors (simulated LiDAR/RaDAR/Cameras) and proprietary 3D point clouds of entire cities. These simulations are focused on improving the perception and decision making abilities of self driving cars.

Communication inherently involves two or more vehicles therefore the scope is naturally larger, on the scope of tens to hundreds of vehicles. This scale comfortably falls into the traditional range of microscopic traffic simulation where PTV Vissim and Eclipse SUMO. SUMO is often used in academic publications while PTV Vissim is often used by traffic professionals at the state and national levels.

PTV Vissim

Vissim does not have any built-in V2X capabilities, however it does have an API/COM interface which can be used to programatically control simulation parameters. This means that external V2X capabilities can be built and integrated into existing Vissim simulations. If you watch the presentation above you will see that V2X capabilities were added to the Karlsruhe traffic simulation that is provided by Vissim as a training example.

Python API

Here is an overview of how the external V2X Python library integrates with Vissim. It is important to note that simulations can be controlled from the Vissim interface or directly from the Python interpreter.

Features

  • Add V2X to existing simulations, including simulations that already use a Python script, with a few lines of code
  • Messages are realistically broadcasted/relayed (drop/loss and delay due to distance and congestion)
  • Define all custom message handling logic in a single project file
  • Allows for co-simulation and external visualization
  • Standardized platooning, dynamic speed harmonization, and collision avoidance routines make common use cases faster to build (not yet added)

Limitations

  • Simulation time step. Microscopic traffic is often simulated at 10 steps per second, or 0.1 second time intervals. This means that dynamic events like automated emergency braking are not suitable for microscopic traffic simulation.
  • Speed. Externally control simulations (simulations started from the Python interpreter) have severe speed limitations. This is a result of how Vissim has engineered its COM interface.
